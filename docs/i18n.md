# Internationalisation du site DROPP

Ce document décrit l'architecture multilingue de **dropp.care** : comment elle
fonctionne, comment ajouter une langue, une clé, un visuel, et comment vérifier
le tout. Il s'adresse à quiconque maintient le site.

## 1. Vue d'ensemble

Le site est **100 % statique**, sans framework et **sans bibliothèque i18n
runtime**. Les pages traduites sont **générées au build** par un script Node
(zéro dépendance npm), à partir de deux sources :

```
src/templates/*.html   ← le HTML, balisé avec des clés {{...}}
locales/{code}/*.json  ← les traductions, une arborescence par langue
        │
        ▼  node scripts/build.js
{code}/index.html, {code}/privacy.html …   ← 14 langues × 4 pages = 56 pages
+ index.html (racine détectrice)  + 404.html  + stubs de redirection  + sitemap.xml
```

**Pourquoi ce choix ?** Le site d'origine était un unique `index.html` avec un
hero WebGL, du scrollytelling et des démos interactives finement réglées. Une
génération statique préserve **exactement** ce rendu, n'ajoute **aucun** poids de
bibliothèque, ne provoque **aucun** flash de mauvaise langue (FOUC), et produit
des URL indexables par langue — idéal pour le SEO et les campagnes publicitaires.

Déploiement : **GitHub Pages** (domaine `dropp.care` via `CNAME`). Le fichier
`.nojekyll` est **obligatoire** — sans lui, Jekyll interpréterait les gabarits
`{{…}}` restés dans le repo. On committe les pages générées (elles sont servies
telles quelles).

## 4. Clés de traduction

Les clés sont **sémantiques et stables** (jamais dérivées de la phrase française),
organisées par section fonctionnelle et par fichier :

- `common.json` — sélecteur de langue, pied de page (partagé par toutes les pages).
- `home.json` — toute la page d'accueil, groupée par section (`hero.*`, `demo.*`,
  `calc.*`, `pricing.*`, `faq.*`…). Le sous-arbre **`home.js.*`** contient les
  chaînes **dynamiques** injectées dans le runtime JS (voir §7).
- `privacy.json`, `support.json`, `legal.json` — les pages légales.

Dans un template, une clé se réfère par son chemin pointé **sans le nom de
fichier** (le fichier est le premier segment) :

```html
<h2>{{home.demo.title}}</h2>
<p>{{home.calc.body}}</p>
<a href="{{path.privacy}}">{{common.footer.privacy}}</a>
```

### Variables « builtin » (fournies par le build, pas par les locales)

`{{builtin.htmlLang}}`, `{{builtin.dir}}`, `{{builtin.canonical}}`,
`{{builtin.hreflangs}}`, `{{builtin.langSwitcher}}`, `{{builtin.footerLangNav}}`,
`{{builtin.ogImage}}`, `{{builtin.filmSrc}}`, `{{builtin.filmPoster}}`,
`{{builtin.i18nJson}}`, `{{path.home|privacy|support|legal}}`. Les liens internes
via `{{path.*}}` sont **localisés automatiquement** (ils pointent vers la bonne
langue) et peuvent être utilisés à l'intérieur d'une valeur de traduction (ex.
`support.q4Body` contient `<a href="{{path.privacy}}">`).

## 5. Ajouter une langue

1. Ajouter une entrée dans `src/site.config.json` → `locales` (code, hreflang,
   htmlLang, dir, name natif, ogLocale, numberLocale).
2. Créer `locales/<code>/` avec les 5 fichiers JSON (copier `locales/fr/` comme
   gabarit, traduire les valeurs — voir `docs/translation-brief.md`).
3. `npm run i18n:check` → corriger jusqu'à **0 erreur**.
4. `npm run build` → la langue apparaît partout (sélecteur, hreflang, sitemap).
5. `npm test`.

Rien d'autre : le sélecteur, les balises `hreflang`, le sitemap et le détecteur
de langue se mettent à jour **automatiquement** à partir de la config.

## 6. Ajouter / modifier une clé

1. Ajouter la clé dans `locales/fr/<fichier>.json` (source) **et** au minimum
   `locales/en/<fichier>.json` (repli).
2. La référencer dans le template : `{{fichier.chemin.de.la.cle}}`.
3. Idéalement, la traduire dans les 12 autres langues. À défaut, le build émet un
   **avertissement de repli** (jamais silencieux) et la page affiche l'anglais.
4. `npm run i18n:check` puis `npm run build`.

## 7. Pluriels, nombres, unités et durées

**Ne jamais concaténer** `count + " gouttes"`. Les chaînes dynamiques passent par
le runtime `window.I18N` (défini dans `src/partials/i18n.js`, alimenté au build
avec **uniquement** les chaînes `home.js.*` de la langue courante) :

```js
I18N.t('rate', {v: 42})          // "42 gtt/min"  (ja/zh : "42 滴/分")
I18N.t('durMLong', {m: 2, n: 2}) // pluriel : "2 minutes" / "2 минуты" / "2分"
I18N.num(1.43, 2, 2)             // "1,43" (fr) · "1.43" (en/ja) — Intl.NumberFormat
```

- **Pluriels** : une clé plurielle est un objet aux formes **CLDR** de la langue
  (`{"one":…,"other":…}` ; ru = `one/few/many/other` ; ar = jusqu'à
  `zero/one/two/few/many/other` ; ja/ko/zh = `other` seul). `I18N.t` choisit la
  forme via `Intl.PluralRules`. `other` est **obligatoire**.
- **Nombres / décimales** : `I18N.num` s'appuie sur `Intl.NumberFormat(locale)` →
  séparateur décimal correct par langue (virgule en fr/de/es…, point en en/ja…).
  Pour l'**arabe**, `numberLocale` vaut `ar-u-nu-latn` : chiffres **latins** (choix
  produit pour la lisibilité médicale).
- **Unités normalisées** : `mL`, `mL/h`, `h`, `min`, `s` restent stables.
- **Gouttes/minute** : unité compacte `gtt / min` conservée pour les langues
  latines + ko/ar/ru ; **localisée en `滴 / 分` pour ja/zh-CN/zh-TW** (voir la
  décision dans `translation-brief.md`). La forme développée (« gouttes/min ») est
  dans la prose de chaque locale.

**Règle d'or** : ne jamais modifier la logique de calcul médical ni les facteurs
de goutte (10/15/20/60) selon la langue. Un test le vérifie.

## 8. RTL (arabe)

- `dir="rtl"` est posé sur `<html>` par le build pour l'arabe.
- La mise en page utilise des propriétés **logiques** (`margin-inline-start`,
  `inset-inline-end`, `text-align:end`…) : elle se mire automatiquement.
- Les valeurs sensibles (nombres, unités, débits, chiffres du cadran) sont
  **isolées en LTR** (`direction:ltr; unicode-bidi:isolate`) pour rester lisibles
  et non ambiguës — voir `src/partials/i18n.css`, section « RTL ».
- Le sélecteur de langue passe naturellement à gauche (inset-inline-end).

Pour vérifier : ouvrir `/ar/`, contrôler l'alignement, les flèches du sélecteur,
les barres de progression, et que « 500 mL », « 22:12 », « 42 » restent lisibles.

## 9. Polices CJK / arabe

Aucune police externe n'est chargée (performance). Des **piles système** dédiées
sont appliquées par `[lang]` dans `i18n.css` :
`Hiragino Sans` (ja), `Apple SD Gothic Neo` (ko), `PingFang SC/TC` (zh),
`SF Arabic`/`Noto Sans Arabic` (ar), avec interlignage et approche ajustés.

## 10. Visuels localisables

| Visuel | Statut | Comment localiser |
|--------|--------|-------------------|
| Maquettes iPhone / Apple Watch, vignettes de démo | **SVG inline** → déjà pilotées par les traductions | Rien à faire, le texte vient des locales |
| `dropp-film.mp4` (+ poster) | **Rastérisé, texte incrusté** | `fr` → film FR ; **toutes les autres** → `dropp-film-en.mp4` (anglais). Mapping dans `site.config.json` → `media`. Repli automatique sur le film FR si l'anglais est absent |
| `og-image.png` | **Rastérisé, texte FR** | Convention prête : déposer `og-image-<code>.png` (ex. `og-image-en.png`) → détecté automatiquement, sinon repli sur `og-image.png` |
| `favicon.png`, `icon.png` | Neutres (logo) | Aucun besoin de version localisée |

**Convention de fichiers par langue** (repli propre si absent) : le build teste
`fs.existsSync` et retombe sur la version par défaut. Pour ajouter un visuel
localisé, il suffit de déposer le fichier à la bonne adresse — aucun code à
changer. Voir la liste des visuels restant à produire dans le rapport de
livraison.

## 11. SEO international

Chaque page génère : `<title>` + meta description traduits, Open Graph
(`og:title/description/locale/image`) et Twitter localisés, `<link rel="canonical">`
propre à la langue, le **jeu complet de `hreflang`** (14 langues + `x-default`),
et des `alt` localisés. Le `sitemap.xml` liste la racine + les 56 pages avec leurs
`xhtml:link rel="alternate"`. `x-default` pointe vers `/` (racine détectrice) pour
l'accueil, vers `/en/…` pour les sous-pages. JSON-LD `MobileApplication` par langue.

## 12. Détection de langue & choix manuel

`root.html` (servi à `/`) exécute un petit script :
1. **choix mémorisé** (`localStorage['dropp-lang']`) — **prioritaire** ;
2. sinon **`navigator.languages`** (jamais de géoloc IP), avec repli intelligent
   (`zh` → `zh-cn`/`zh-tw`, `pt` → `pt-br`/`pt-pt`, base de langue) ;
3. sinon **anglais**.
Puis `location.replace('/<code>/' + search + hash)` — **les paramètres UTM et
l'ancre sont préservés**. Un `<noscript>` renvoie vers `/en/`.

Le clic sur une langue du sélecteur **mémorise le choix** et **reporte
`?utm_*` + `#ancre`** vers la page cible (`src/partials/i18n.js`).

## 13. Campagnes publicitaires par langue

Pointer chaque campagne directement vers sa landing localisée :
`https://dropp.care/en/?utm_source=instagram&utm_campaign=launch_en`. Les
paramètres survivent aux changements de langue. Checklist dédiée dans le rapport
de livraison et §« Lancer une pub » ci-dessous.

## 14. Vérifier, construire, tester

```bash
npm run i18n:check   # cohérence des traductions (clés, placeholders, pluriels, marque…)
npm run build        # génère les 56 pages + racine + 404 + stubs + sitemap
npm test             # i18n:check + build + suite node:test (16 tests)
npm run serve        # sert le site localement sur http://localhost:4173
npm run dev          # build + serve
```

`i18n:check` signale : clés manquantes, clés inconnues/inutilisées, traductions
vides, placeholders `{x}` incohérents, formes plurielles invalides, marque DROPP
altérée, balises `<b>/<br>/<a>` déséquilibrées, locale non déclarée. **Il ne masque
jamais une erreur** : un repli est un avertissement visible pendant le dev.

## 15. Contenus exigeant une validation humaine

Les traductions cliniques, avertissements et mentions légales sont classés par
sensibilité dans [`docs/translation-review.md`](translation-review.md). Les textes
de **niveau 3** (avertissements cliniques, responsabilité, confidentialité) doivent
être **relus par un professionnel de santé natif ou bilingue** avant diffusion
dans un nouveau marché.
